# OmegaClaw Core v0.01a Technical Walkthrough

This walkthrough is for a technical AGI reviewer who wants to understand what changed in v0.01a, why it changed, how it is implemented, and how confident we are that it works. It replaces older ad hoc review notes and reflects the current private readiness branch.

## One-Sentence Position

OmegaClaw v0.01a turns the original MeTTa loop into a more usable persistent cognitive substrate: the LLM remains a cognition provider, while memory, skills, spaces, modules, trace, reasoning, and action boundaries are inspectable symbolic machinery.

## Chronological Development Summary

1. **Syntax membrane hardening** — command parsing moved from brittle parenthesis balancing toward signature-aware repair and fail-closed syntax errors.
2. **Runtime spaces** — persistent, agenda, beliefs, world, events, activity, cleanup, attention, skills, skill-triggers, scratch, scratch-ttl, assume, and module spaces became registered names rather than guessed raw handles.
3. **Skill affordance directory** — skills gained symbolic cards, topics, arguments, effects, risks, preferred-use hints, aliases, and recall triggers.
4. **Input-aware context routing** — fresh input can recall relevant memories and skill hints, while still requiring deliberate query/space inspection for confident answers.
5. **Context payload compaction** — bulky write payloads are mechanically omitted from the LLM-facing view while raw history remains exact on disk.
6. **Continuity and energy posture** — `pin`, `energy-status`, `set-energy-mode`, `cycle-status`, and restart/reboot affordances were clarified as volatile continuity and operating posture, not durable memory.
7. **Symbolic cleanup affordances** — memory cleanup moved toward proposal/commit and exact atom/id workflows rather than prose reconstruction or hidden deletion.
8. **Module boundary** — optional abilities moved under `modules/*` with module manifests, entries, signatures, catalogs, affordances, skills, dependencies, and trace declarations.
9. **AgentVerse module** — remote-agent discovery, registration, listener, inbox, calls, and traces became module-owned instead of hardcoded core behavior.
10. **Assume/FabricPC module** — assumption-making became an inspectable MeTTa atomspace plus numeric FabricPC mirror/daemon for prediction, audit, learning, and symbolic writeback proposals.
11. **Channel, sense, media, and body-app modules** — WhatsApp, Telegram, Mattermost, web control, web search, glucose, senses, image/video, Game Boy, Codex-code, scheduler, VM policy, and related surfaces were normalized as modules.
12. **Tests, benchmarks, demos, and review gates** — the branch now includes unit tests, smoke classification, baseline benchmark comparison, demo results, patch audit, and name/security review notes.

## Major Improvements

### Syntax Membrane

**What it does:** Turns model output into explicit MeTTa skill calls, lowers rich text writes to base64-safe forms, and rejects malformed command shapes with recoverable syntax errors.

**How implemented:** `src/helper_command_parser.py` reads `SkillSignature`, `SignatureLowering`, `SignatureSpace`, and recovery declarations from core/module signature files. The parser remains a membrane: it parses, normalizes, and validates command shapes; it does not decide cognition.

**Why:** Continuous agents fail often at the boundary between prose and action. A reliable substrate needs errors to be visible and recoverable rather than silently misexecuted.

**What is right:** The surface is symbolic and extensible by module metadata. Large multiline writes use byte-preserving base64 lowering. Bad arity/types fail closed.

**What is still wrong or limited:** It is still a parser for an LLM-shaped command stream, not a full verified programming language. Good skill cards and examples still matter.

**Confidence:** High for covered command surfaces. Unit tests and syntax smoke tests exercise common and previously fragile shapes.

### Runtime Spaces And Memory Boundary

**What it does:** Exposes named spaces for long-lived cognition: `persistent`, `agenda`, `beliefs`, `world`, `events`, `activity`, `cleanup`, `attention`, `skills`, `skill-triggers`, `scratch`, `scratch-ttl`, `assume`, and module-owned spaces such as `agentverse`.

**How implemented:** `skills_runtime_spaces.metta` registers named spaces, aliases, persistence paths, and pressure limits. Python helpers only create safe runtime files when MeTTa declarations ask for them.

**Why:** Raw `&space` handles are not a stable interface for an agent. Named spaces let Omega inspect, query, transform, persist, and reason without guessing implementation handles.

**What is right:** The policy lives in MeTTa atoms. Runtime persistence is explicit and ignored by Git. Generic space tools can work across modules.

**What is still wrong or limited:** Some same-space cleanup workflows remain young and need more live operational mileage. Chroma recall can still miss memories due phrasing/similarity limits.

**Confidence:** Medium-high for registration/persistence; medium for autonomous cleanup ergonomics.

### Skill Affordance Directory

**What it does:** Makes skills discoverable as symbolic attention objects rather than hidden Python routing. Omega can ask what skills exist, what they are for, what arguments they take, and when to use them.

**How implemented:** Affordance atoms live in core and module `affordance.metta`/catalog files: `Skill`, `SkillTopic`, `SkillArg`, `SkillCardLine`, `PreferredWhen`, `Risk`, `Effect`, triggers, aliases, and context hints.

**Why:** A persistent agent with many abilities needs a small visible top layer and a queryable deep catalog. Otherwise it forgets tools it has or invents tools it does not.

**What is right:** The catalog is inspectable, removable, and module-owned. Input-aware skill recall can surface relevant hints without Python choosing the action.

**What is still wrong or limited:** Skill cards require maintenance. If a card is stale or too terse, the agent can still misuse a valid tool.

**Confidence:** High for structure; medium-high for card quality after the current sweep.

### Context Payload Compaction

**What it does:** Omits only bulky payload arguments from the LLM-facing history view when the owning skill declares a compact-payload policy. Raw history remains exact.

**How implemented:** `SkillContextView` and `SkillContextPolicy` atoms declare compaction behavior. `src/helper_metta.py` mechanically replaces the large command expression with a placeholder such as `<context-omitted-payload chars=N raw-history-preserved>` in the context view only.

**Why:** Generated HTML, files, and base64 payloads can consume the entire attention window and hide the actual cognition around them.

**What is right:** No semantic summary is made. The raw trace is the source of truth. The demo suite shows a sanitized 67,802-char raw history becoming a 750-char prompt view while before/after thought atoms stay visible.

**What is still wrong or limited:** Reviewers must understand that this is a context view policy, not deletion. More real-history-shaped fixtures can be added over time.

**Confidence:** High. Unit tests and demo results cover the policy boundary.

### Module System

**What it does:** Turns non-core abilities into portable organs with explicit contracts.

**How implemented:** Modules provide `module.toml`, `entry.metta`, `signatures.metta`, `catalog.metta`, `affordance.metta`, `skills.metta`, and optional `src/`. The active composition is controlled by `modules/loader.metta`.

**Why:** AGI systems need capabilities to be inspectable and removable. Hidden Python imports blur identity, action, memory, and provider boundaries.

**What is right:** Module dependencies and secrets are declared. Skill surfaces disappear when modules are not loaded. Tests enforce contract shape.

**What is still wrong or limited:** Some modules are proof-of-concept or optional-body surfaces. `body_container` and `omega_vm` are marked POC and should not be presented as mature cognition.

**Confidence:** High for the contract; mixed by module maturity.

### AgentVerse

**What it does:** Lets Omega discover, register, call, and receive messages from remote uAgents through a symbolic module surface.

**How implemented:** `modules/agentverse` owns bridge Python, listener, registry, trace, signatures, skills, and cards. Calls and inbox entries are trace-backed under module runtime state.

**Why:** Remote agents should be skills available to the substrate, not hardcoded alternate cognition providers.

**What is right:** The module surface is explicit and optional. Live testing demonstrated remote end-to-end call behavior through the listener path. The demo suite verifies the module surface without a network call.

**What is still wrong or limited:** Network availability and remote-agent behavior are outside core control. Public demos should avoid requiring third-party live endpoints.

**Confidence:** Medium-high for module mechanics; medium for remote ecosystem reliability.

### Assume / FabricPC

**What it does:** Gives Omega an assumption space: she can create situations, features, actions, edges, outcomes, predictions, audits, growth pressure, and reviewed writebacks.

**How implemented:** `&assume` remains canonical symbolic state. FabricPC runs as a numeric mirror/daemon over those atoms and returns predictions, audits, learning deltas, mutation evidence, and writeback atoms. Python performs parsing, numeric calculation, daemon transport, validation, and trace writes; it does not own the graph's meaning.

**Why:** Assumption-making is neither pure symbolic deduction nor opaque neural inference. Omega needs a space where uncertain guesses can be made, audited, learned from, and revised visibly.

**What is right:** The demo graph is ordinary MeTTa-shaped data. Learn/writeback produces symbolic deltas and reviewable mutation atoms. Canonical mutations are explicit.

**What is still wrong or limited:** FabricPC dependency setup is optional and can skip in clean environments. More real domains are needed before broad claims.

**Confidence:** Medium-high for demo path and unit-covered mechanics; medium for generality until more domains are exercised.

### Tests, Benchmarks, And Demos

**What it does:** Separates correctness gates, MeTTa smoke classes, benchmark metrics, and reviewer demos.

**How implemented:** Unit tests cover parser, modules, memory/runtime helpers, bridges, Assume, AgentVerse, and contracts. `tests/run_metta_smokes.py` classifies isolated/runtime/live-risk smokes. `docs/review/benchmark_suite.py` compares against ASI Alliance `origin/main`. `docs/review/demo_suite.py` prints deterministic scenario evidence.

**Why:** Reviewers need to know whether a claim is unit-tested, smoke-tested, benchmarked, live-tested, or only documented.

**What is right:** Current local gate reports 279 unit tests OK with 15 skips; review audit passes; demo suite reports 6/6 demos passing.

**What is still wrong or limited:** Some full-runtime MeTTa skill behavior still requires live runtime testing because isolated PeTTa execution does not always reduce imported skill definitions.

**Confidence:** High for local gates; medium for optional live services.

## Important Cautions

- OmegaClaw is not a finished AGI and should not be represented as one.
- LLM premise formulation remains the largest correctness risk.
- Formal truth values are only as good as the atoms supplied to the engine.
- Optional modules can expose real-world side effects; run them only with appropriate sandboxing and credentials.
- Live deployments may contain private runtime memory, channel traces, and local web state that are intentionally outside this repo.

## Bottom Line

v0.01a is a credible out-of-box research substrate, not a final product. Its strongest contribution is architectural hygiene: cognition is kept in MeTTa-visible spaces and symbolic affordances, while Python/JS remain membranes for IO, transport, parsing, numeric mirrors, and trace. That makes the system inspectable enough for serious AGI review and modular enough for upstream/core maintainers to take pieces selectively.
